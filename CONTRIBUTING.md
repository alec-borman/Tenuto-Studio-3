# Contributing to Tenuto Studio 3.0

Thank you for your interest in the Tenuto project. By contributing, you aren't just helping build a music tool; you are helping engineer the **Definitive System of Record for Musical Intent.**

Because Tenuto is a high-performance, cross-language protocol (Rust, Wasm, TypeScript), we maintain a high standard for architectural integrity. Please read this guide to understand our "Black Box" philosophy before submitting a Pull Request.

---

## 🏛️ The Architectural Philosophy

Tenuto is built on the **Strict Decoupling of Intent from Manifestation.**

1.  **Logic is Sovereign:** The Rust compiler (`tenutoc`) is the source of truth. It should never know about pixels, speakers, or browsers. It only knows about mathematical ratios and musical symbols.
2.  **Statelessness:** The presentation layers (WebGL/Audio) must be "blind." They simply observe the Intermediate Representation (IR) generated by the compiler and render it.
3.  **Deterministic Math:** We avoid floating-point numbers in the core logic whenever possible. We use a **Rational Temporal Engine** (ticks and fractions) to prevent "drift" in long compositions.

## 🛠️ Development Setup

### Prerequisites
* **Rust Toolchain:** (latest stable) for the compiler and daemon.
* **Node.js 18+:** for the IDE and presentation engine.
* **Wasm-Pack:** To bridge the Rust core into the browser.
* **Ableton Link (Optional):** If you are working on the `tenutod` synchronization layer.

### The Build Pipeline
1. **Clone & Install:**
   ```bash
   npm install
   ```
2. **Build the Logic Core:**
   ```bash
   cd tenutoc
   wasm-pack build --target web --out-dir ../public/pkg
   ```
3. **Launch the Laboratory:**
   ```bash
   npm run dev
   ```

## 🛡️ The "Titanium Core" Testing Rules

We enforce **Strict Test-Driven Development (TDD).** We do not accept features without proof of structural integrity.

### 1. Lexical and Parser Tests (Rust)
If you add a new keyword or musical symbol, you must add a unit test in `tenutoc/src/lexer.rs` or `parser.rs`. We utilize `proptest` for property-based fuzzing.
```bash
cd tenutoc
cargo test
```
*Your code must pass a minimum of 10,000 fuzzing iterations before a PR is reviewed.*

### 2. Semantic Analysis (TypeScript)
If you add a new musical rule (e.g., "cannot have a 13th interval in this mode"), add it as a Linter rule in `src/compiler/linter.ts`.
```bash
npm run test
```

## 📬 Contribution Workflow

### Step 1: Open an Architectural RFC
For major changes, open an Issue titled `[RFC] Your Feature Name`. Explain the **shape of the data** before you explain the implementation. We prioritize data integrity over UI "flashiness."

### Step 2: The "Small Batch" PR
Keep PRs atomic. Do not mix a change to the Rust Lexer with a change to the WebGL particle colors.
* **Logic PRs:** Focus on `tenutoc`.
* **Visual PRs:** Focus on `src/engraver`.
* **Acoustic PRs:** Focus on `src/audio`.

### Step 3: Zero-Warning Policy
We do not merge code with warnings. 
* **Rust:** Must pass `cargo clippy`.
* **TypeScript:** Must pass `npm run lint`.

## 🗝️ Core Domains

* **`tenutoc/`**: The "Brain." Custom LL(1) parser using `chumsky`. If you change this, you are changing the language's grammar.
* **`src/engraver/` (TEAS)**: The "Eyes." Implements 1D skylines and Bezier pathing. This handles collisions of musical ink.
* **`src/audio/`**: The "Ears." Manages the `AudioWorklet` and standardizes the hardware handoff.
* **`tenutod/`**: The "Nervous System." Orchestrates the TEDP (Tenuto Execution & Delegation Protocol) over OSC and Link.
